Overall conclusion
- 当前版本功能覆盖很高，但服务端安全边界、容器校验策略、跨维度/区块生命周期管理仍有高风险点。
- 若要稳定兼容 JEI / Effortless，并支持后续旧版本 port，建议先完成一轮“安全与平台抽象”重构。

High-risk findings (P0)

[P0-1] 服务端缺少 RTS 活跃态与半径校验，存在远程交互面
- 线索
  - RtsStorageManager.placeSelected:226
  - RtsStorageManager.breakPlaced:365
  - RtsStorageManager.pickupLinkedToCarried:419
  - RtsStorageManager.returnCarriedToLinked:317
  - 以上路径没有统一校验“玩家当前是否处于 RTS 活跃会话 + 是否在 anchor 半径内”。
- 风险
  - 修改客户端可直接发包在已加载区块进行 place/break/import/pickup。
  - 结合 chunk loader 可能形成跨区块远程操作面（含破坏与资源迁移）。
- 建议
  - 新增统一守卫：`validateRtsAction(player, targetPos)`。
  - 至少校验：RtsCameraManager 会话活跃、目标维度一致、距离 anchor <= MAX_RADIUS、chunk loaded、权限可交互。

[P0-2] linked storage 仅存 BlockPos，不含维度
- 线索
  - Session.linkedPositions 为 `List<BlockPos>`（RtsStorageManager:831 附近）。
  - resolveLinkedHandlers 使用 `player.serverLevel()` 当前维度解析（RtsStorageManager:741）。
- 风险
  - 切维度后重进 RTS，可能把旧维度链接位置解释为新维度同坐标方块。
  - 造成错误链接、错误取料/存料，甚至被利用为跨维度语义漏洞。
- 建议
  - 链接键升级为 `(ResourceKey<Level>, BlockPos)`。
  - requestPage / place / break / pickup / return 全部按维度过滤有效链接。

[P0-3] 通过反射放宽菜单校验，兼容脆弱且有越权面
- 线索
  - relaxOpenedMenuValidation:631
  - 将 `ContainerLevelAccess` 改为 `NULL`，并包装 `Container.stillValid` 始终 true。
- 风险
  - 距离/上下文校验被绕过，modded 菜单在不同版本字段变化时易崩。
  - 容易引入“远距离持续打开容器”类漏洞。
- 建议
  - 逐步移除通用反射 patch。
  - 改为白名单策略（仅对确认安全的菜单类型）或显式 RTS 容器桥接层。

[P0-4] RTS break 追踪边界过宽，且 refund 语义与真实掉落不完全一致
- 线索
  - RtsBlockTrackingEvents.onEntityPlace:19 / onEntityMultiPlace:30 标记所有玩家放置。
  - breakPlaced:365 只要 tracker 命中即可 insta-break。
  - refundFromState 基于方块状态简单退款（RtsStorageManager:687），非完整掉落语义。
- 风险
  - 非 RTS 放置方块也可被 RTS 快速破坏。
  - 某些方块/容器/NBT 场景可能出现收益偏差（dup/loss 风险都存在）。
- 建议
  - tracker 区分“RTS 放置”与“普通放置”，并记录 owner UUID。
  - break 走统一掉落路由（尽量接近原版 drop pipeline），再吸入 linked。

Medium-risk findings (P1)

[P1-1] 多链接去重缺失，同一底层库存可能重复计数
- 线索
  - resolveLinkedHandlers 逐 BlockPos 拉 handler（RtsStorageManager:741），无 endpointKey 去重。
- 风险
  - 双箱/网络端点多入口场景，UI 计数重复，导致误导与失败重试。
- 建议
  - 引入 `endpointKey` 去重层，聚合统计前先去重。

[P1-2] 容器 overlay 的“来源追踪”仅 itemId，关闭时自动归还可能误判
- 线索
  - pendingOverlayCarriedItemId（RtsClientInputGate:39）
  - onScreenClosing 自动 return（RtsClientInputGate:271）
- 风险
  - 玩家手上同 itemId 但非 overlay 来源物品，可能被误回收到 linked。
- 建议
  - 用 pickup token（UUID/nonce）做来源标记，而非仅 itemId。

[P1-3] 缺少服务端限流与抗重放
- 线索
  - place/break/pickup/return 入口无每 tick 限流。
- 风险
  - 高频包可导致服务器压力飙升，且增加竞态概率。
- 建议
  - 每玩家动作 token bucket（place/break 与 page 请求分开限速）。

[P1-4] session 生命周期清理不足
- 线索
  - onRtsDisabled 只保留会话（RtsStorageManager:55）。
  - 仅看到登出/换维 stop camera（RtsbuildingMod:129/136），未见 storage session 清理策略。
- 风险
  - 长服内存累积、旧链接残留、跨场景状态污染。
- 建议
  - 登出清理或落盘（可配置）；至少增加 TTL 清理。

Lower-risk findings (P2)

[P2-1] 容器内容抽取仅覆盖 `Container`，未覆盖纯 IItemHandler 方块实体
- 线索
  - extractContainerContents:597 只判断 `instanceof Container`。
- 影响
  - 某些 mod 机器破坏吸收时可能漏回收。
- 建议
  - 增加 `Capabilities.ItemHandler.BLOCK` 路径抽取。

[P2-2] requestPage 每次全量扫描所有 linked 槽位，规模上升后有性能压力
- 线索
  - requestPage:89 内部全量聚合 counts。
- 建议
  - 做 snapshot cache + dirty 标记，或按页增量查询。

Compatibility considerations

JEI
- 现状
  - RTS 启用时全局取消交互键映射（RtsClientInputGate.onInteractionKey:55）。
  - 容器界面会叠加 RTS linked overlay 并拦截点击/滚轮。
- 兼容风险
  - JEI 容器扩展区域可能与 RTS overlay 操作区域冲突。
  - 输入事件处理链可能互相吞事件。
- 建议
  - 加 `compat.jei` 模式：容器打开且 JEI active 时，RTS overlay 位置避让或可一键隐藏。
  - 将交互键 cancel 收敛到“BuilderScreen/世界交互区”而非宽泛全局。

Effortless Building
- 现状
  - RTS 使用 RMB 点击交互 + RMB 拖拽旋转 + MMB 拖拽平移 + 滚轮缩放。
- 兼容风险
  - 与 Effortless 常用键位/拖拽语义冲突概率高。
- 建议
  - 加 `compat.effortless` 键位仲裁层（RTS active 时可自动切换按键上下文）。
  - 保持“RTS 只走 vanilla use pipeline，不直接调用 Effortless 内部交互”。

Chunkload / exploit-focused considerations
- 重点不是“未加载区块”，而是“已加载远程区块”。
- 当前有 `hasChunkAt` 校验，但缺少“是否在 RTS anchor 半径内”与“是否 RTS active”双重门。
- 结合 chunk loader 或其他玩家加载区块时，可能形成远程放置/破坏/物品迁移通道。
- 建议把 `anchor + radius + active session + dimension` 作为所有 C2S 动作硬前置条件。

Player death / lifecycle considerations
- 当前仅看到 logout 与 changed-dimension 处理，未见 death/respawn 专门逻辑。
- 建议补齐
  - 死亡时强制 stop RTS。
  - 清理客户端 pending overlay token 与临时 carried 语义状态。
  - linked 保留策略做配置项：`keep_links_on_death`。
  - 若保留 linked，重生后先做维度/区块可用性复检再启用。

Extensibility and old-version port (重点)

Current coupling hotspots
- 网络：CustomPacketPayload + PayloadRegistrar（NeoForge 1.21 风格）。
- 能力：Capabilities.ItemHandler.BLOCK 直接散落在核心流程。
- 输入/屏幕：大量 NeoForge Screen/Input 事件直接耦合。
- 菜单：反射改内部字段，版本敏感度极高。

建议的抽象边界（先在 1.21.1 内完成）
- `IRtsActionGuard`：统一动作安全校验（active/radius/dimension/permission/chunk）。
- `IStorageEndpointResolver`：按平台解析与去重 endpointKey。
- `IMenuInteractBridge`：容器打开/校验策略隔离，避免核心写反射。
- `INetworkBridge`：消息 DTO 与平台发送注册分离。
- `IClientInputBridge`：输入事件与 UI 交互桥接，避免 BuilderScreen 过重。

Port path (recommended)
1. 在当前分支先做“安全守卫 + 维度化链接 + 去反射”三件事。
2. 抽出 platform 接口层（不改玩法逻辑，只挪胶水代码）。
3. 开 `forge-1.19.2` 分支，只替换 platform 实现：
   - 网络注册层
   - capability lookup 层
   - client event/input 层
   - 菜单互操作层
4. 用同一套协议 DTO 和同一套服务端规则测试集回归。
